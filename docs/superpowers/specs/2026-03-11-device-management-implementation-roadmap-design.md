# 设备管理后端项目实施路线设计

## 1. 背景

当前仓库仍处于 Spring Boot 初始化骨架阶段，仅包含启动类、基础配置和上下文加载测试；而 `AGENTS.md` 已经定义了完整的设备管理后端目标形态，包括技术栈、目录结构、分层约定、模块边界、定时任务和接口映射。

当前“初始化骨架”的判断依据如下：

- 仅存在启动类 `src/main/java/com/jhun/backend/DeviceManagementBackendApplication.java`
- 仅存在上下文加载测试 `src/test/java/com/jhun/backend/DeviceManagementBackendApplicationTests.java`
- 仅存在最小配置文件 `src/main/resources/application.properties`
- `pom.xml` 仍为初始化依赖集合，尚未补齐 `AGENTS.md` 中规划的安全、数据访问、缓存、JWT、AI 等核心能力
- 当前目录结构尚未进入 `AGENTS.md` 约定的 `config/common/dto/entity/mapper/service/controller/scheduler` 模块化形态

因此，本次设计不直接拆到单个类或单个文件的实现步骤，而是先制定一份可长期执行的总实施计划文档，作为项目从 0 到 1 的统一开发路线图。该路线图必须满足以下要求：

- 严格依据 `AGENTS.md` 进行模块与功能拆分
- 按阶段逐步落地，避免并行开发导致基础能力反复返工
- 每完成一部分功能或模块，必须执行代码审查和功能测试
- 每个阶段都形成可验收、可回归、可进入下一阶段的闭环

## 2. 设计目标

本设计用于指导后续产出 1 份总实施计划文档，覆盖以下内容：

- 以项目总体路线图的形式组织开发工作，而不是零散的模块任务列表
- 计划粒度控制在“模块 -> 功能点 -> 关键实现项 -> 测试项 -> 审查项”
- 兼顾项目管理视角与开发执行视角，既能说明先后顺序，也能指导每阶段落地
- 在文档中内置固定的质量门禁，确保阶段完成标准清晰可执行

## 3. 关键约束与默认假设

### 3.1 约束

- 以 `AGENTS.md` 的技术栈与架构约定为基准
- 采用三层架构：`Controller -> Service -> Mapper`
- 使用 `DTO/VO` 进行输入输出隔离
- 使用统一响应体与统一异常处理机制
- 采用分阶段实施，不允许核心依赖尚未稳定时提前实现下游模块
- 每个阶段必须包含代码审查与功能测试门禁

### 3.2 默认假设

- 阶段 0 需要先把当前初始化工程调整到符合 `AGENTS.md` 的目标基线
- 当前代码包结构 `com.jhun.backend` 与 `AGENTS.md` 约定的包路径不一致，计划默认在早期阶段完成包结构与目录规范对齐
- 当前依赖仍是最小骨架，后续阶段会逐步补齐 Spring Security、MyBatis-Plus、MySQL、Redis、JWT、Spring AI 等能力
- 当前设计聚焦“总实施路线图”，不直接展开为具体类级或方法级开发清单

## 4. 计划文档结构设计

后续实施计划文档采用单一总文档结构，统一包含全项目路线图，并按阶段组织内容。

文档层级如下：

`项目阶段 -> 模块 -> 功能点 -> 关键实现项 -> 测试项 -> 审查项 -> 完成标准`

每个阶段固定包含以下栏目：

- 目标
- 范围
- 前置依赖
- 关键实现项
- 交付物
- 代码审查清单
- 功能测试清单
- 完成标准
- 进入下一阶段前检查项

该结构的目的，是让计划文档既能作为项目总路线图，也能直接作为阶段执行清单使用。

此外，后续总实施计划必须把“阶段说明”与“质量门禁”强绑定：每个阶段都必须显式写出里程碑目标、入口条件、审查清单、测试清单、退出条件，避免只保留阶段标题和概述说明。

## 5. 实施阶段划分

### 5.1 阶段 0：项目骨架与工程规范

建立项目底盘，统一目录结构、依赖基线、配置体系、基础响应与异常、公共常量与枚举、测试基座、数据库与缓存接入策略。

该阶段不追求业务价值闭环，而是为后续全部业务模块提供统一复用能力，避免后续模块重复建设基础设施。

阶段 0 的最小完成面至少包括：

- 包路径、目录结构与模块命名对齐 `AGENTS.md`
- Spring Boot、Spring Security、MyBatis-Plus、MySQL、Redis、JWT、Spring AI 等基础依赖基线确认
- `application.yml` 与多环境配置骨架建立
- 统一响应体、统一异常处理、公共枚举、公共常量、基础工具类落位
- 持久层、缓存层、鉴权层和测试基座的最小可运行能力建立
- 邮件通知基础配置、通知记录基础模型与调度任务基础支撑预留

### 5.2 阶段 1：用户与权限

实现注册、登录、刷新 Token、验证码、密码重置、用户管理、角色权限、冻结解冻等能力。

该阶段是后续业务访问控制的前置依赖，设备、预约、借还、AI 等模块都依赖统一身份体系和权限校验。

### 5.3 阶段 2：设备与分类

实现设备分类管理、设备 CRUD、设备图片上传、设备状态管理和状态变更记录。

该阶段输出统一的设备主数据模型，为预约、借还与统计提供基础业务对象。

### 5.4 阶段 3：预约管理

实现预约申请、审核、冲突检测、签到、人工处理、自动过期与提醒相关能力。

该阶段构成核心业务主流程，是借还管理与逾期管理的上游依赖。

### 5.5 阶段 4：借还管理

实现借用确认、归还确认、借还历史和设备状态联动。

该阶段必须建立在预约流程稳定的基础上，确保状态流转与业务规则一致。

### 5.6 阶段 5：逾期管理

实现逾期识别、限制策略、解除限制、通知与逾期统计口径。

该阶段依赖借还记录与用户状态体系，属于主流程后的派生治理能力。

### 5.7 阶段 6：智能对话

实现 AI 对话、意图识别、槽位提取、提示模板管理与对业务服务的调度集成。

该阶段放在核心业务稳定后推进，避免 AI 层反向驱动核心业务边界。

### 5.8 阶段 7：统计分析

实现概览统计、设备利用率、借用统计、热门时段和预聚合任务。

统计模块依赖前序业务数据沉淀，因此安排在业务主体能力之后实施。

### 5.9 阶段 8：联调、质量加固与发布准备

进行跨模块联调、异常链路回归、配置分环境整理、性能与安全检查、文档补全和发布前验收。

该阶段用于统一收口，确保系统达到可交付状态。

### 5.10 横切能力归属

为避免总实施计划出现模块无归属的问题，横切能力按以下方式纳入阶段：

- 通知能力基础设施（邮件配置、通知记录、模板支撑）在阶段 0 建立底座
- 用户侧通知（验证码、密码重置、冻结解冻通知）归属阶段 1
- 预约审核、签到、即将开始、审核超时等通知归属阶段 3
- 逾期提醒与限制解除通知归属阶段 5
- `TokenCleanupProcessor`、`SessionTimeoutProcessor` 等系统运维任务优先在阶段 1 建立，与认证会话体系一起落地
- 其余跨模块调度联调、容错和运行验收在阶段 8 统一收口

## 6. 阶段依赖原则

整体实施顺序遵循以下原则：

`基础设施 -> 身份权限 -> 主数据 -> 主流程 -> 派生流程 -> 智能增强 -> 统计分析 -> 联调发布`

采用该顺序的原因如下：

- 先稳定底层规范和公共能力，降低返工成本
- 先完成认证与权限，保证后续接口和业务规则具备统一鉴权基础
- 先建设设备主数据，再建设预约与借还流程，确保业务对象统一
- 将 AI 与统计放在核心业务稳定后推进，避免过早引入复杂依赖
- 让每个阶段都具备独立测试和验收边界

## 7. 质量门禁设计

### 7.1 代码审查机制

每完成一个阶段或一个独立功能点，必须执行正式代码审查，审查维度至少包括：

- 是否遵循 `Controller -> Service -> Mapper` 分层
- 是否正确使用 `DTO/VO`
- 是否统一使用响应封装与异常处理机制
- 权限控制、鉴权链路和用户状态校验是否完整
- 事务边界、状态流转与定时任务幂等性是否合理
- 命名、目录、模块边界是否符合 `AGENTS.md`
- 公共枚举、常量、工具类是否复用，是否出现重复实现

代码审查流程固定为：

`开发完成 -> 自检清单 -> 代码审查 -> 修复问题 -> 回归验证`

阻塞级问题至少包括以下类型：

- 编译失败或测试无法运行
- 鉴权缺失、越权访问、用户冻结状态绕过
- 设备、预约、借还、逾期等核心状态流转错误
- 事务边界不当导致数据不一致
- 定时任务非幂等、重复执行风险明显
- 接口协议与 `AGENTS.md` 约定严重不一致

### 7.2 功能测试机制

每个阶段都必须规划对应测试集合，至少包括：

- 单元测试
- 集成测试
- 接口功能测试
- 异常场景测试

核心链路需要同时覆盖成功路径与失败路径，重点包括：

- 登录与权限校验
- 设备状态流转
- 预约冲突检测与审核
- 借用确认与归还确认
- 逾期识别与限制释放
- AI 对话调用与异常降级
- 统计聚合结果校验

测试执行基线应在实施计划中固化为可复用命令模板，至少包括：

- 开发中针对单个模块运行 `mvn test -Dtest=指定测试类或方法`
- 阶段完成时运行 `mvn clean test`
- 涉及跨模块集成、配置变更、定时任务或发布验收时运行 `mvn clean verify`
- 如已接入质量检查插件，则追加运行 `mvn checkstyle:check`、`mvn spotbugs:check`

后续实施计划需要明确：每个阶段至少新增或更新哪些单元测试、哪些集成测试、哪些接口验证项，以及哪些测试结果是进入下一阶段的硬门槛。

### 7.3 阶段完成标准

一个阶段只有在以下条件全部满足后，才允许进入下一阶段：

- 计划范围内功能点开发完成
- 代码审查通过，阻塞级问题已修复
- 阶段测试全部通过
- 必要文档与配置同步更新
- 下阶段所需依赖条件已经满足

其中“阶段测试全部通过”默认指：

- 与本阶段功能直接相关的单元测试通过
- 与本阶段功能直接相关的集成测试通过
- 阶段验收命令达到预期通过结果
- 不存在阻塞后续阶段的已知高风险缺陷

## 8. 实施计划文档的输出要求

基于本设计生成的实施计划文档应满足以下要求：

- 采用单一总文档形式，完整覆盖项目全部阶段
- 每个阶段都细化到模块与功能点，而不是仅列出标题
- 每个阶段都必须显式写出代码审查与功能测试安排
- 文档可被后续开发过程直接作为里程碑执行清单使用
- 文档内容应优先面向“逐阶段落地”，而不是一次性大爆炸开发

## 9. 风险与注意事项

- 当前仓库与目标架构差距较大，阶段 0 的基线统一非常关键
- 若早期未统一包结构、依赖基线和公共规范，后续模块会发生大面积返工
- 预约、借还、逾期三者属于强耦合业务链，必须严格按顺序推进
- AI 模块必须在核心业务服务稳定后再接入，否则接口编排与意图映射会频繁变动
- 统计分析必须依赖前序业务数据模型稳定，否则口径难以统一

## 10. 结论

本设计确定采用“1 份总实施计划文档 + 分阶段里程碑推进 + 阶段性代码审查与功能测试门禁”的组织方式，用于指导设备管理后端项目从初始化骨架逐步演进到完整业务系统。

后续实施计划将以本设计为依据，重点解决阶段顺序、模块拆分、功能覆盖、质量门禁和阶段交付标准五个方面的问题。
